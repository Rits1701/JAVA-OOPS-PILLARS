4 OOP Pillars — Java Interview Guide

Code examples · Interview Q&A · Violation traps · One-liners to memorize

🔒
Encapsulation
Bundling data (fields) and the methods that operate on that data into a single unit (class), and restricting direct access to internal state. External code interacts only through a well-defined public interface.
"Hide the data, expose the behaviour. The class is the only guardian of its own state."
Private fields
All fields are private. State is never directly accessible from outside.
Controlled access
Getters/setters (or better: behaviour methods) enforce validation and business rules.
Immutability
No setters at all = strong encapsulation. State set once at construction time.
Tell, Don't Ask
Ask the object to do something; don't ask for its data and decide yourself.
Violation vs Correct
❌ Broken encapsulation
class BankAccount {
    public double balance; // public field!
    public String owner;
}

// Caller can do ANYTHING — no rules:
account.balance = -9999;   // no validation
account.balance += 1000;   // logic scattered
account.owner = null;       // corrupt state
✅ Proper encapsulation
class BankAccount {
    private final String owner;
    private double balance;

    public BankAccount(String owner, double initial) {
        if (initial < 0) throw new IllegalArgumentException();
        this.owner   = Objects.requireNonNull(owner);
        this.balance = initial;
    }

    public void deposit(double amount) {
        if (amount <= 0) throw new IllegalArgumentException();
        balance += amount;
    }

    public void withdraw(double amount) {
        if (amount > balance) throw new InsufficientFundsException();
        balance -= amount;
    }

    public double getBalance() { return balance; }
}
Strongest encapsulation — Java record (immutable)
// Java 16+ record — all fields private final, no setters ever
record Money(BigDecimal amount, Currency currency) {

    // Compact constructor: enforce invariants at creation time
    Money {
        Objects.requireNonNull(amount,   "amount required");
        Objects.requireNonNull(currency, "currency required");
        if (amount.compareTo(BigDecimal.ZERO) < 0)
            throw new IllegalArgumentException("amount cannot be negative");
    }

    // Behaviour method — returns new instance (object never mutates)
    public Money add(Money other) {
        if (!currency.equals(other.currency))
            throw new CurrencyMismatchException();
        return new Money(amount.add(other.amount), currency);
    }
}
Interview trap: "Isn't encapsulation just getters and setters?" — No. A class with public setters for every field has no real encapsulation. The goal is enforcing invariants. Prefer behaviour methods (deposit(), withdraw()) over naked getters/setters. Mention the Tell, Don't Ask principle.
Interview Q&A
Encapsulation is about bundling data and the methods that operate on it into a class and exposing only what's necessary through a controlled interface. Data hiding is one mechanism — using private access modifiers. Together they ensure: (1) objects maintain valid state at all times, (2) implementation can change without breaking callers, (3) business rules live in one place rather than scattered across the codebase.
Yes — and it's often better without them. The ideal is behaviour methods: account.deposit(100) instead of account.setBalance(account.getBalance() + 100). A class that exposes a setter for every field has no real encapsulation — anyone can bypass invariants. Immutable classes (using record, final fields, no setters) represent the strongest form of encapsulation.
Tell an object what to do; don't Ask for its data and then decide yourself. Bad: if (account.getBalance() >= amount) account.setBalance(...). Good: account.withdraw(amount) — the account enforces its own rule. This keeps logic co-located with data and prevents duplication of business rules across callers.
Maintenance: implementation details (switching from ArrayList to LinkedList internally) don't affect callers — only the public interface matters. Testing: you test through behaviour methods, not internal state. You test that account.withdraw() throws when overdrawn — you don't assert on the raw balance field. State changes are an implementation detail.

Inheritance
A mechanism allowing a subclass to acquire the properties and behaviours of a superclass, enabling code reuse and establishing an IS-A relationship between types.
"A Dog IS-A Animal. Use inheritance when the IS-A relationship is genuinely true and stable."
extends
Single inheritance only. Java avoids the diamond problem this way.
super keyword
Call parent constructor or parent method from the subclass.
@Override
Redefine parent behaviour in child. Compile-time safety net.
IS-A test
Before inheriting, ask: "is a Dog always an Animal in every context?" If yes, inherit.
Core hierarchy example
abstract class Animal {
    private final String name;
    private final int    age;

    public Animal(String name, int age) {
        this.name = name;
        this.age  = age;
    }

    public String getName() { return name; }
    public int    getAge()  { return age;  }

    // Concrete method — shared by all animals
    public void eat() { System.out.println(name + " is eating"); }

    // Abstract — every subclass MUST provide its own version
    public abstract void makeSound();

    public String describe() { return "I am " + name + ", aged " + age; }
}

// Dog IS-A Animal ✓
class Dog extends Animal {
    private final String breed;

    public Dog(String name, int age, String breed) {
        super(name, age);   // parent constructor must run first
        this.breed = breed;
    }

    @Override
    public void makeSound() { System.out.println(getName() + ": Woof!"); }

    @Override
    public String describe() {
        return super.describe() + ", breed: " + breed; // extends parent
    }

    public void fetch() { System.out.println("Fetching!"); } // Dog-specific
}

class Cat extends Animal {
    public Cat(String name, int age) { super(name, age); }

    @Override
    public void makeSound() { System.out.println(getName() + ": Meow!"); }
}

The famous mistake — Inheritance vs Composition
❌ Wrong: Stack extends Vector
// Real JDK mistake — java.util.Stack
// Stack IS-NOT-A Vector in behaviour.
// Vector.add(0, element) bypasses LIFO.
class Stack<T> extends Vector<T> {
    // inherits 40+ irrelevant methods
    // caller can insert at any index!
    // breaks the Stack contract
}
✅ Right: Composition (HAS-A)
// Stack HAS-A Deque internally
class Stack<T> {
    private final Deque<T> storage = new ArrayDeque<>();

    public void push(T item) { storage.push(item); }
    public    T pop()        { return storage.pop(); }
    public    T peek()       { return storage.peek(); }
    // Only 3 methods exposed. Contract respected.
}
Interview gold: Always mention "Favour Composition Over Inheritance" (Effective Java, Item 18). Inheritance creates tight coupling — child breaks when parent changes. The classic gotcha: java.util.Stack extends Vector — a design mistake in the JDK itself. Use inheritance only when the IS-A relationship is genuinely stable and the subclass truly substitutes for the parent everywhere.
Interview Q&A
The diamond problem: if class C extends both A and B, and both define method(), which does C inherit? Ambiguity. Java resolves this by allowing only single-class inheritance. Multiple interface implementation is allowed. With Java 8 default methods a diamond can appear — the class must explicitly override the conflicting method and can call a specific parent's version via Interface1.super.method().
Overriding (instance methods): resolved at runtime based on actual object type — this is runtime polymorphism. @Override is your compile-time safety net.

Hiding (static methods): resolved at compile time based on reference type. If parent and child both declare the same static method and you call it via a parent reference, the parent version runs — even if the object is a child. Static methods are NOT polymorphic.
Constructors are NOT inherited. The child must define its own and use super() to call the parent's. If you omit super(), Java calls the no-arg parent constructor implicitly — if none exists, compilation fails.

Private methods are NOT inherited — they belong only to the declaring class and are invisible to subclasses. Declaring the same signature in a subclass creates an independent new method, not an override.
🎭
Polymorphism
The ability of an object to take many forms. The same method call behaves differently based on the actual object type. Two flavours: compile-time (overloading) and runtime (overriding via dynamic dispatch).
"One interface, many implementations. The caller doesn't need to know which one runs."
Compile-time
Method overloading — same name, different signatures. Resolved by compiler.
Runtime
Method overriding — subclass redefines parent method. JVM picks via vtable at runtime.
Dynamic dispatch
JVM resolves the call based on the actual object type, not the reference type.
Liskov (LSP)
Subtype must be substitutable for its supertype without breaking correctness.
Part 1 — Runtime Polymorphism (most important for interviews)
interface Shape {
    double area();
    double perimeter();
    default void describe() {
        System.out.printf("Area=%.2f, Perimeter=%.2f%n", area(), perimeter());
    }
}

class Circle implements Shape {
    private final double radius;
    public Circle(double r) { radius = r; }
    @Override public double area()      { return Math.PI * radius * radius; }
    @Override public double perimeter() { return 2 * Math.PI * radius; }
}

class Rectangle implements Shape {
    private final double width, height;
    public Rectangle(double w, double h) { width = w; height = h; }
    @Override public double area()      { return width * height; }
    @Override public double perimeter() { return 2 * (width + height); }
}

// ── RUNTIME POLYMORPHISM — same call, different execution ──
List<Shape> shapes = List.of(
    new Circle(5), new Rectangle(4, 6), new Triangle(...)
);

shapes.forEach(Shape::describe);      // JVM picks correct area() at runtime

double totalArea = shapes.stream()
    .mapToDouble(Shape::area)           // no instanceof, no casting — clean!
    .sum();
Part 2 — Compile-time Polymorphism (overloading)
class Printer {
    // Same name — different parameter types = overloading
    public void print(String s)             { System.out.println("String: " + s); }
    public void print(int n)                { System.out.println("Int: "    + n); }
    public void print(double d)             { System.out.println("Double: " + d); }
    public void print(String s, int copies) { /* ... */ }
}

Printer p = new Printer();
p.print("hello");  // compiler picks print(String)  at compile time
p.print(42);      // compiler picks print(int)     at compile time
p.print(3.14);   // compiler picks print(double)  at compile time

// JDK example: System.out.println is overloaded 10 times
Spring Boot daily usage — polymorphism in action
interface PaymentGateway {
    PaymentResult charge(Amount amount, Card card);
}

@Component("stripe")    class StripeGateway   implements PaymentGateway { /* ... */ }
@Component("braintree") class BraintreeGateway implements PaymentGateway { /* ... */ }
@Component("mock")      class MockGateway      implements PaymentGateway { /* tests */ }

@Service
class PaymentService {
    private final PaymentGateway gateway;

    public PaymentService(@Qualifier("stripe") PaymentGateway gateway) {
        this.gateway = gateway;
    }

    public PaymentResult processPayment(Amount a, Card c) {
        return gateway.charge(a, c);  // polymorphic — swap gateway without touching this class
    }
}
Key distinction: Overloading = compile-time, resolved by parameter types. Overriding = runtime, resolved by actual object type. Interviewers love: "Can you override a static method?" — No, static methods are hidden not overridden. Also know covariant return types: overriding method can return a subtype of the parent's declared return type (since Java 5).
Interview Q&A
Dynamic dispatch is the runtime selection of method implementation based on actual object type. The JVM uses a vtable (virtual method table) — a per-class array of method pointers. When you call shape.area(), the JVM looks up area in the vtable of the actual object (e.g., Circle), not the declared reference type (Shape). The JIT compiler optimises this with inline caching, so the overhead is negligible in practice.
Static methods cannot be overridden — only hidden. Resolution happens at compile time based on reference type. @Override on a static method won't compile.

Private methods cannot be overridden either — they're not visible to subclasses. The same signature in a subclass is a brand-new independent method. No runtime dispatch happens.
LSP: if S is a subtype of T, objects of type T may be replaced with S without breaking correctness. Polymorphism only works correctly when LSP holds. Classic violation: Square extends Rectangle — setting width on a Square must also change height, breaking the Rectangle caller's assumption that dimensions are independent. Fix: use a common Shape interface. If your subclass needs to throw exceptions or weaken preconditions, it's an LSP violation.
OCP: open for extension, closed for modification. Without polymorphism: if (shape instanceof Circle) ... else if (shape instanceof Rectangle) ... — every new shape requires modifying existing code. With polymorphism: define a Shape interface with area(), and adding a Pentagon requires zero changes to existing code. Callers using the Shape abstraction get the new behaviour for free.

Abstraction
Hiding implementation complexity and exposing only what is relevant to the caller. Define what something does — not how it does it. Achieved through interfaces and abstract classes.
"A driver uses the steering wheel without knowing the steering rack mechanics. That's abstraction."
Interface
Pure contract — what to do. Any unrelated class can implement it. Multiple allowed.
Abstract class
Partial implementation — what + some how. Forces subclasses to fill the gaps.
Reduce coupling
Callers depend on the abstraction, not the concrete type — easy to swap implementations.
vs Encapsulation
Encapsulation hides internal state. Abstraction hides implementation complexity.
Interface vs Abstract Class — the decisive comparison
interface — pure contract
interface Notification {
    // All methods implicitly public abstract
    void send(String message, String recipient);

    // Default method (Java 8+) — optional override
    default void sendWithRetry(String msg, String to) {
        for (int i = 0; i < 3; i++) {
            try { send(msg, to); return; }
            catch (Exception e) { /* retry */ }
        }
    }

    // Any unrelated class can implement:
}

class EmailNotification implements Notification { /* SMTP */   }
class SmsNotification   implements Notification { /* Twilio */ }
class SlackNotification implements Notification { /* Webhook */}
class PushNotification  implements Notification { /* FCM */    }
abstract class — template method
abstract class BaseNotification {
    // Shared state — interfaces can't have fields
    private final String    senderName;
    private final RateLimit rateLimiter;

    protected BaseNotification(String sender) {
        this.senderName  = sender;
        this.rateLimiter = new RateLimit(100);
    }

    // Template Method — skeleton fixed here
    public final void send(String msg, String to) {
        rateLimiter.checkLimit();         // shared
        String formatted = format(msg);  // shared
        doSend(formatted, to);            // ← abstract hook
    }

    protected abstract void doSend(String msg, String to);

    private String format(String msg) {
        return "[" + senderName + "] " + msg;
    }
}

class SmtpNotification extends BaseNotification {
    @Override
    protected void doSend(String msg, String email) {
        smtpClient.send(email, msg); // SMTP-only details
    }
}
Abstraction vs Encapsulation — the most confused pair
// ABSTRACTION = hiding COMPLEXITY (what, not how)  ← design level
// ENCAPSULATION = hiding INTERNAL STATE (data)     ← implementation level

interface EmailService {
    void sendEmail(String to, String subject, String body);
    // ↑ ABSTRACTION: caller has no idea if this is SMTP, SES, SendGrid, or a mock
}

class SmtpEmailService implements EmailService {
    private final SmtpClient client;   // ← ENCAPSULATION: hidden field
    private int sentCount = 0;         // ← ENCAPSULATION: hidden state

    @Override
    public void sendEmail(String to, String sub, String body) {
        client.connect();              // ← ABSTRACTION: all SMTP complexity hidden
        client.authenticate();
        client.send(to, sub, body);
        client.disconnect();
        sentCount++;                    // ← ENCAPSULATION: state change internal
    }
}

// Caller sees NEITHER the implementation detail (abstraction)
// NOR the SmtpClient or counter (encapsulation)
EmailService email = new SmtpEmailService(client);
email.sendEmail("bob@co.com", "Hi", "Hello!");
When to choose — definitive rule
// ── Use INTERFACE when ────────────────────────────────────
// • Unrelated classes need the same capability
// • Multiple "inheritance" needed (class can implement many)
// • Pure contract with no shared state
// • Building a plugin/extension point
// JDK: Comparable, Runnable, Iterable, Serializable

// ── Use ABSTRACT CLASS when ───────────────────────────────
// • Shared instance FIELDS across a family
// • Template Method Pattern (skeleton + variant steps)
// • All subclasses are the same IS-A family
// JDK: AbstractList, HttpServlet, AbstractQueuedSynchronizer

// ── Best practice: both together ─────────────────────────
interface Repository<T, ID> { /* public contract */ }

abstract class AbstractRepository<T, ID> implements Repository<T, ID> {
    // shared: caching, auditing, pagination logic
}

class OrderRepository extends AbstractRepository<Order, UUID> {
    // only order-specific queries here
}
One-liner that impresses: "Abstraction operates at the design level — what interface do I expose? Encapsulation operates at the implementation level — how do I protect my internal state?" Example: List is the abstraction (callers don't care if it's ArrayList or LinkedList), and ArrayList's private Object[] elementData is encapsulation.
Interview Q&A
Interface: pure contract — no instance fields (only static final constants), all methods public. A class can implement many. Use when unrelated classes need the same capability. From Java 8+: default and static methods allowed. From Java 9+: private methods for internal reuse.

Abstract class: partial implementation — can have instance fields, constructors, protected methods. A class can extend only one. Use for the Template Method Pattern or when you need to share state and code across a tightly related family.

Rule: prefer interfaces for flexibility. Use abstract classes when you genuinely need shared state or behaviour across a family.
Yes, abstract classes can and should have constructors — called via super() from the subclass constructor, essential for initialising shared fields.

No, you cannot directly instantiate an abstract class — new AbstractAnimal() is a compile error. You instantiate a concrete subclass, and the chain of super() calls runs the abstract class constructor up the hierarchy.
Before Java 8, adding a method to an interface broke all existing implementors — a massive backwards-compatibility problem. Default methods allow interfaces to evolve without breaking existing code. Example: Collection.stream() was added as a default in Java 8 — ArrayList, HashSet, etc. got it for free.

Diamond with defaults: if a class implements two interfaces with conflicting defaults, it must explicitly override and can delegate via Interface1.super.method().
Defines the skeleton of an algorithm in an abstract class — invariant steps are implemented, variant steps are left abstract for subclasses. The calling method is typically final so subclasses can't break the skeleton. Heavy JDK usage: HttpServlet.service() calls doGet()/doPost(). Spring's JdbcTemplate handles connection lifecycle while you supply just the query via a callback. Abstract class + Template Method + hook methods is one of the most powerful patterns for reusable frameworks.
